# PRD Orchestrator スキル

PRD作成フロー全体を制御するオーケストレーションスキル。
ユーザーの入力から深度を判定し、適切なフローに誘導する。

## 深度自動判定ロジック

ユーザーの最初の発言を以下の基準で判定する:

| 深度 | 判定条件 | 例 |
|------|---------|-----|
| Minimal | 要件が具体的で明確。修正箇所が特定されている | 「〇〇画面のバグを直すIssue作って」「typo修正」 |
| Standard | 機能追加・改善だがスコープが1機能に収まる | 「〇〇にフィルター機能を追加したい」 |
| Comprehensive | 要件が抽象的。複数機能にまたがりそう。非エンジニアからの依頼 | 「〇〇みたいな機能がほしい」「新しい課金モデルを導入したい」 |

### 判定指標（5軸スコアリング）

各指標を 1〜3 で採点し、合計で深度を決定する:

| 指標 | 1（Minimal寄り） | 2（Standard寄り） | 3（Comprehensive寄り） |
|------|-----------------|-------------------|----------------------|
| 具体性 | 画面名・機能名・エラー内容など固有名詞あり | 機能カテゴリは特定されている | 抽象的な表現のみ |
| スコープ | 単一画面・単一要素 | 単一機能（複数画面可） | 複数機能にまたがる |
| 技術的明確性 | 技術的解決策が明示 | 方向性はあるが詳細未定 | 技術的手段が未定 |
| ステークホルダー数 | 1者のみ | 2者 | 3者以上 |
| 抽象度 | 具体的な指示 | 要望レベル | 「みたいな」「的な」「っぽい」等の曖昧表現 |

**合計スコア判定:**
- 5〜7: Minimal
- 8〜11: Standard
- 12〜15: Comprehensive

## 判定後のフロー

### Minimal → AI DLC案内

```
この内容はスコープが明確なため、PRDなしでAI DLCに直接進められます。

PRDなしで進める方法:
1. Claude Code で直接タスクを依頼してください（例: 「〇〇画面の△△を修正して」）
2. 必要に応じてリポジトリパスを指定し、Claude Code にコード変更を依頼できます
3. Issue作成が必要な場合は「〇〇のIssueを作って」と依頼してください
```

### Standard → PRD推奨

```
「PRDの作成をお薦めします。作成しますか？」
→ Yes: PRD作成フロー開始
→ No: PRDなしで進める方法を案内（上記 Minimal のテキスト参照）
```

### Comprehensive → PRD必須

```
「この規模にはPRDが必要です。作成しますか？」
→ Yes: PRD作成フロー開始
→ No: PRDなしで進める方法を案内（以下のリスク注記付き）

⚠️ 注意: この規模の要件をPRDなしで進めると、以下のリスクがあります:
- スコープクリープによる開発遅延
- ステークホルダー間の認識齟齬
- 手戻りコストの増大
- 品質基準の不明確化
```

## PRD作成フロー（Yes の場合）

### Step 1: ドメイン知識収集
- ユーザーに対象プロダクトのリポジトリパスと走査対象（機能領域やキーワード）を確認する（任意。「なし」「スキップ」も許容）
- `domain-collector` サブエージェントを起動
- 渡すコンテキスト:
  - ユーザーの元の発言
  - 対象プロダクト名（デフォルト: ランサーズ）
  - 対象機能の推定カテゴリ
  - リポジトリパス（ユーザーが提供した場合のみ）
  - 走査対象（ユーザーが指定した機能領域やキーワード）
- 受け取るもの:
  - ドメインサマリー（300字以内）
  - 更新されたKnowledgeファイルパス一覧
  - 機能コンテキスト要約
  - エビデンス情報（根拠の種類・内容・画像パス）

#### Gate 1→2: ドメイン知識の充足確認
以下をすべて満たすまでStep 2に進まない:
- [ ] _product.yaml に対象プロダクトの基本情報（ペルソナ・KPI・競合）が存在する
- [ ] _feature.yaml が作成/更新され、現行フロー or 課題が1つ以上記載されている
- [ ] domain-collector の unanswered_questions が空、または残存質問をユーザーが「後で回答」と明示した

**次アクション提示（AskUserQuestionで表示）:**
- A) PRDヒアリングへ進む（推奨）
- B) 収集したドメイン知識を確認・修正してから進む
- C) ドメイン情報をさらに追加する

### Step 2: PRD要素ヒアリング（Orchestrator自身がAskUserQuestionで実行）

AskUserQuestion を使用して、4回のCallで構造化ヒアリングを実施する。

#### 自動推定と質問スキップの原則

Step 1 で収集したドメイン知識（OST構造・エビデンス・known_issues）と、ユーザーの初期入力から、各質問の回答を推定する。

- **確信度が高い → 自動で埋めて質問をスキップ** する（ユーザーに確認しない）
- **確信度が低い → 確認パターン**（domain-interviewer/SKILL.md「既存データ確認パターン」参照）で推定値を提示し、ユーザーに選んでもらう
- **推定できない → 通常の質問** として提示する
- 1つの Call 内で全問がスキップされた場合、その Call 自体を省略する

#### 選択肢生成の原則

スキップされなかった質問の選択肢は、以下の優先順位で動的に生成する:

1. **OST構造から生成**（最優先）: ost_structure の opportunities / solutions / high_uncertainty_items から具体的な選択肢を導出
2. **Knowledge動的**（次点）: _product.yaml / _feature.yaml のフィールドから生成
3. **汎用フォールバック**（最後）: 上記で選択肢が不足する場合のみ静的選択肢を補完

選択肢は必ずドメインコンテキストに即した具体的な表現にする。抽象的なカテゴリ名ではなく、Step 1 で得た固有名詞・具体的課題・ソリューション名を使う。

#### Call P-1: 仮説確認（Who, Pain, Gain）— 3問

| # | 質問 | header | 選択肢ソース | multiSelect |
|---|------|--------|-------------|-------------|
| Who | この機能の主要ターゲットは？ | ターゲット | **Knowledge動的**（_product.yamlのペルソナから最大4つ） | true |
| Pain | 現在のペインポイントは？ | ペイン | **OST優先**（ost_structure.opportunities から最大4つ）。不足時は _feature.yaml の known_issues で補完。それでも不足時は汎用フォールバック | true |
| Gain | 期待される効果は？ | 期待効果 | **OST優先**（ost_structure.desired_outcome + solutions から期待効果を導出、最大4つ）。不足時は汎用: 効率化 / 品質向上 / 収益増加 / ユーザー定着 | true |

※ エビデンスは Step 1（domain-collector）の B-1b で収集済み。未収集の場合のみ、P-1 の後にフォローアップでエビデンスの有無を確認する。

#### Call P-2: スコープ・KPI — 4問

| # | 質問 | header | 選択肢ソース | multiSelect |
|---|------|--------|-------------|-------------|
| Phase1 | Phase1で作る範囲は？ | Phase1範囲 | **OST優先**（ost_structure.solutions から優先度の高いソリューション群をスコープ候補として最大4つ提示）。不足時は汎用: MVP最小 / コア＋補助 / フル実装 / 段階リリース | false |
| Out | スコープ外とするものは？ | スコープ外 | **Knowledge動的**（_feature.yaml の related_features + OST で優先度が低いソリューションから最大4つ）。不足時は汎用: 管理画面 / 外部連携 / 多言語・国際化 / 特になし | true |
| KPI | 成功を測るKPI指標は？ | 成功KPI | **Knowledge動的**（key_metrics + ost_structure.desired_outcome から最大4つ） | true |
| Target | 目標水準は？ | 目標水準 | **エビデンス参照**（evidence_items にデータ・数値がある場合、現在値からの改善目標を提示）。不足時は汎用: xx%改善 / 絶対値目標 / 定性目標 / 後で設定 | false |

#### Call P-3: ユーザーストーリー・要件 — 3問

| # | 質問 | header | 選択肢ソース | multiSelect |
|---|------|--------|-------------|-------------|
| UC | 主要なユースケース種別は？ | ユースケース | **Knowledge動的**（_product.yaml のペルソナ × OST の opportunities からユースケースを導出、最大4つ）。不足時は汎用: 新規ユーザー / 既存ヘビー / 管理者・運営 / エッジケース | true |
| Must | Must要件は？ | Must要件 | **OST優先**（ost_structure.solutions のうち high_uncertainty でないものから最大4つ）。不足時は improvement_hypotheses → 汎用フォールバック | true |
| Want | Want要件は？ | Want要件 | **OST優先**（ost_structure.solutions のうち優先度が低いもの、または high_uncertainty_items から最大4つ）。不足時は汎用: UI/UX改善 / 自動化 / 通知・連携 / なし | true |

- UC: preview でユーザーストーリーテンプレート形式（「〇〇として、〇〇したい。なぜなら〇〇だから。」）を表示

#### Call P-4: リスク・撤退基準 — 3問

| # | 質問 | header | 選択肢ソース | multiSelect |
|---|------|--------|-------------|-------------|
| Risk | 想定されるリスクカテゴリは？ | 想定リスク | **OST優先**（ost_structure.high_uncertainty_items からリスク候補を導出、最大4つ）。不足時は汎用: 技術リスク / ユーザー受容 / ビジネス影響 / 運用負荷 | true |
| Mitigate | リスク軽減策は？ | リスク軽減 | **OST優先**（high_uncertainty_items.suggested_experiment から軽減策候補を導出、最大4つ）。不足時は汎用: A/Bテスト / フィーチャーフラグ / 手動運用併用 / ロールバック計画 | true |
| Exit | 撤退を判断する基準は？ | 撤退条件 | **KPI連動**（P-2 の KPI 回答から撤退条件候補を生成）。不足時は汎用: KPI未達 / 障害・品質 / ユーザー反応 / コスト超過 | true |

#### Gate 2→3: ヒアリング完了確認
以下をすべて満たすまでStep 3に進まない:
- [ ] P-1〜P-4 の全Callが完了し、各Callで1つ以上の選択/回答がある
- [ ] ターゲットユーザー（Who）が1つ以上特定されている
- [ ] 成功KPIが1つ以上設定されている

**次アクション提示（AskUserQuestionで表示）:**
- A) 競合・市場リサーチへ進む（推奨）
- B) ヒアリング回答を見直して修正する
- C) リサーチをスキップしてPRD生成へ直接進む（Minimal/Standard向け）

### Step 3: リサーチ実行
- `research-analyst` サブエージェントを起動
- 渡すコンテキスト:
  - Knowledge の競合情報ファイルパス
  - ドメインサマリー（Step 1の出力）
  - 対象機能名と概要
  - ヒアリングで得た仮説
- 受け取るもの:
  - 競合分析サマリー
  - 市場動向サマリー
  - 差別化機会
  - 補足根拠（supplementary_evidence）— ヒアリングのペイン・課題を裏付けるリサーチデータ

#### Gate 3→4: リサーチ充足確認
以下をすべて満たすまでStep 4に進まない:
- [ ] 競合分析サマリーが返却されている（空文字列でない）
- [ ] 差別化機会が1つ以上特定されている
- 注: market_summary が空でもブロックしない（市場データが取得できないケースは許容）

**次アクション提示（AskUserQuestionで表示）:**
- A) PRD生成へ進む（推奨）
- B) リサーチ結果を確認してから進む
- C) 追加リサーチを実行する（特定の競合・市場セグメントを深掘り）

### Step 4: PRD生成
- `prd-writer` サブエージェントを起動
- 渡すコンテキスト:
  - Knowledge ファイルパス一覧
  - ドメインサマリー
  - ヒアリング回答（構造化済み）
  - リサーチサマリー
  - エビデンス情報（根拠の種類・内容・画像パス）
  - リサーチ補足根拠（supplementary_evidence）— research-analyst が発見した補足データ
- 受け取るもの:
  - PRD本体（outputs/{機能名}_{YYYY-MM-DD}.md）
  - 品質チェック結果
  - [要確認] 項目一覧

#### Gate 4→5: PRD品質確認
以下をすべて満たすまでStep 5に進まない:
- [ ] prd-writer の quality_check.passed が true（またはスコアが全項目数の80%以上）
- [ ] PRDファイルが outputs/ に正常出力されている
- [ ] [要確認] 項目が10件以下である（多すぎる場合は情報不足のためStep 2に戻る）

### Step 5: レビューと修正
- 生成されたPRDをユーザーに提示
- 修正指示を受け付け、prd-writerに再委譲
- 完了後、PRD要約をknowledge/に保存

### Step 6: Notion連携（オプション）
- Notion MCPが利用可能な場合:
  - 「NotionにPRDをアップロードしますか？」と確認
  - Yesの場合: Notion MCPツールを使用してPRDをアップロード
    - 新規: 指定ページ配下に新規ページとしてPRDを作成
    - 更新: 既存ページを検索し、見つかれば内容を更新
  - Noの場合: 「後から /upload-notion コマンドでアップロードできます」と伝える
- Notion MCPが利用不可の場合: このステップをスキップ

## AIPO設計思想の適用

### Jeff Patton の PO思考プロセス対応

本フローはAIPO（miyatti777/aipo）の Sense → Focus → Discover → Deliver サイクルに対応:

| Phase | AIPO | PRD Orchestrator | ポイント |
|-------|------|-------------------|---------|
| **Sense** | Goal設定・コンテキスト収集 | Step 1: ドメイン知識収集 | 既存Knowledgeの確認と不足分の補完 |
| **Focus** | 調査とGoal分解 | Step 2: PRD要素ヒアリング | ユーザーからの要件を構造化 |
| **Discover** | 計画・コマンド生成 | Step 3: リサーチ実行 | 競合・市場の調査で仮説を検証 |
| **Deliver** | タスク実行・成果物生成 | Step 4: PRD生成 | 全インプットを統合しPRDを出力 |

### HITL（Human In The Loop）統合

各Stepでユーザーの確認・判断ポイントを明確に設定:
- Step間の遷移時に中間成果物をユーザーに提示
- 「確認なしで次に進む」ことはしない
- ユーザーの修正指示は即座に反映

### 再帰分解（Fractal Decomposition）

Comprehensive判定時、要件が大きすぎる場合はAIPOのSubLayer分解を参考に:
- 1つのPRDでカバーしきれない場合は、機能ごとに分割PRDを提案
- 各分割PRDは独立して作成・管理可能
- 親PRD → 子PRD の関係性をKnowledgeに記録
