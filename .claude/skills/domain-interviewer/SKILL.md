# ドメインインタビュアー スキル

ドメイン知識のヒアリング質問票と進め方を定義する。
2段階ヒアリング方式（プロダクトレベル → 機能レベル）で構造化されたドメイン知識を収集する。

## 基本原則

1. **既存データ活用:** Knowledgeに既存データがある項目は確認パターン（後述）で確認のみ
2. **AskUserQuestion使用:** 最大4問ずつ AskUserQuestion で提示（1回の呼び出しで最大4問）
3. **選択式優先:** 各問に2〜4個の選択肢を設定（「Other」は AskUserQuestion が自動追加するため明示不要）
4. **ペルソナ意識:** Teresa Torres の Opportunity Solution Tree の構造を参考に、ユーザーの機会（Opportunity）からソリューションへ分解する

## 初期入力からの推定（Pre-fill）

ヒアリング開始前に、ユーザーの初期入力（要望テキスト）を解析し、各質問の回答を推定する。

### 推定ルール

1. **初期入力を各質問の選択肢と照合** し、回答を推定する
2. **確信度が高い場合 → 自動で埋めて質問をスキップ** する（ユーザーに確認しない）
3. **確信度が低い（曖昧・複数候補あり）場合 → 確認パターンで提示** し、ユーザーに選んでもらう
4. **推定できない場合 → 通常の質問** として提示する

### 確信度の判定基準

| 確信度 | 条件 | 処理 |
|--------|------|------|
| 高 | 初期入力に明確なキーワードがあり、選択肢と1対1で対応する | 自動スキップ |
| 低 | キーワードはあるが複数候補に該当する、または間接的な表現 | 確認パターン |
| なし | 手がかりがない | 通常の質問 |

### 推定対象の例

| 質問 | 初期入力の手がかり | 推定結果 |
|------|-------------------|---------|
| F1（機能カテゴリ） | 「検索」「レコメンド」「マッチング」等のキーワード | マッチング・検索 |
| F1 | 「メッセージ」「チャット」等のキーワード | コミュニケーション |
| F1 | 「決済」「支払い」「請求」等のキーワード | 取引・決済 |
| F3（課題カテゴリ） | 「精度が低い」「結果が悪い」等の表現 | 品質・精度 |
| F3 | 「遅い」「重い」等の表現 | パフォーマンス |
| F3 | 「使いにくい」「わかりにくい」等の表現 | UX・操作性 |
| F5（競合実装） | 「AIを導入」「海外では〜」等の表現 | 競合が先行 |
| F6（過去の経緯） | 「以前検討した」「見送った」等の表現 | 検討して見送り |

### 推定の制約

- 1つの Call 内で全問が自動スキップされた場合、その Call 自体を省略する
- 自動スキップした項目は domain_summary の `auto_filled` フィールドに記録する（後工程で参照可能）
- Phase B だけでなく Phase A の質問にも同じ推定ルールを適用する

## 既存データ確認パターン

Knowledgeに既存データがある項目、または初期入力から推定できた項目は、通常の質問ではなく以下の確認パターンに変換する:
- **preview:** 推定した回答内容 or 前回の回答内容を表示
- **選択肢:** そのまま / 一部修正 / 大幅変更 / スキップ
- 「一部修正」「大幅変更」選択時はフォローアップで詳細を確認

## Phase A: プロダクトレベルヒアリング

`knowledge/{product}/_product.yaml` の既存データを確認し、空欄の項目のみ質問する。

### Call A-1: ビジネスモデル基盤（D1, D2, D3）— 3問

AskUserQuestion で以下の3問を一括提示:

| # | 質問 | header | 選択肢 | multiSelect |
|---|------|--------|--------|-------------|
| D1 | プロダクトの主要なビジネスモデルは？ | ビジネスモデル | 手数料型 / サブスク型 / ハイブリッド / フリーミアム | false |
| D2 | クライアント側の主なペルソナは？ | クライアント像 | SMB経営者 / 大企業部門担当 / スタートアップ / 個人事業主 | true |
| D3 | ランサー（ワーカー）側の主なペルソナは？ | ランサー像 | 副業ワーカー / 専業フリーランス / 法人チーム / 学生初心者 | true |

### Call A-2: 指標・競合（D4, D5, D6, NSM）— 4問

AskUserQuestion で以下の4問を一括提示:

| # | 質問 | header | 選択肢 | multiSelect |
|---|------|--------|--------|-------------|
| D4 | 主要KPI指標は？ | KPI指標 | 取引系(GMV/件数/ARPU) / ユーザー活性(MAU/リピート率) / 顧客満足(NPS/CSAT) / 収益性(売上/粗利/LTV) | true |
| D5 | 主な競合プロダクトは？ | 競合 | クラウドワークス / ココナラ / 海外勢(Upwork/Fiverr) / 隣接競合(タイミー等) | true |
| D6 | プロダクト固有の重要用語は？ | 固有用語 | 取引形式の用語 / ユーザー種別の用語 / 課金の用語 / 直接入力したい | true |
| NSM | NSM（North Star Metric）として最重視する指標は？ | NSM指標 | GMV最重視 / 成約件数最重視 / MAU最重視 / リピート率最重視 | false |

- D6: preview で既存登録済み用語があれば表示する

### Call A-2b（条件付き）: 海外競合の詳細 — 1問

D5 で「海外勢(Upwork/Fiverr)」を選択した場合のみ実施:

| # | 質問 | header | 選択肢 | multiSelect |
|---|------|--------|--------|-------------|
| D5b | 特に意識している海外競合は？ | 海外競合 | Upwork / Fiverr / Freelancer.com / Toptal | true |

## Phase B: 機能レベルヒアリング

対象機能のフォルダ・ファイルがなければ新規作成を提案する（ユーザー許諾必須）。

### Call B-1: 機能の位置づけ（F1, F2, F3）— 3問

AskUserQuestion で以下の3問を一括提示:

| # | 質問 | header | 選択肢 | multiSelect |
|---|------|--------|--------|-------------|
| F1 | 今回対象の機能カテゴリは？ | 機能カテゴリ | マッチング・検索 / コミュニケーション / 取引・決済 / ユーザー基盤(プロフィール/オンボーディング/管理) | false |
| F2 | この機能の現状のユーザーフローは？ | 現状フロー | シンプル直線型 / 分岐型 / マルチステップ型 / 双方向型 | false |
| F3 | 既知の課題カテゴリは？ | 課題カテゴリ | UX・操作性 / 品質・精度 / パフォーマンス / ビジネス指標 | true |

- F2: preview で既存フローデータがあれば表示する
- F3: 選択後、Call B-1b でフォローアップ

### Call B-1b（条件付き）: 課題の深掘りとエビデンス — 3問

F3 で課題カテゴリを選択した後に実施:

| # | 質問 | header | 選択肢 | multiSelect |
|---|------|--------|--------|-------------|
| F3b | 選択した課題の深刻度は？ | 課題深刻度 | 致命的 / 重要 / 改善希望 / 仮説段階 | false |
| F3c | 挙げた課題を裏付ける根拠はありますか？ | 根拠の種類 | データ・数値あり / スクリーンショット・画像あり / ユーザーの声・定性情報あり / 根拠なし（仮説段階） | true |
| F3d | 根拠の共有方法は？ | 根拠の共有 | 今すぐ共有したい / 後で共有する | false |

- 「今すぐ共有したい」選択時 → フォローアップで自由入力（URL、ファイルパス、テキスト）を求める
- 「根拠なし（仮説段階）」のみ選択時 → F3d をスキップ
- F3c で「根拠なし（仮説段階）」と他の根拠種類が同時選択された場合 → 根拠ありを優先し、「根拠なし」は無視する

### Call B-2: 関連情報・履歴（F4, F5, F6）— 3問

AskUserQuestion で以下の3問を一括提示:

| # | 質問 | header | 選択肢 | multiSelect |
|---|------|--------|--------|-------------|
| F4 | 関連する他の機能は？ | 関連機能 | **Knowledge動的生成**（既存featureフォルダから最大4つ選出） | true |
| F5 | 競合はこの機能をどう実装していますか？ | 競合実装 | 基本同等 / 競合が先行 / 当社が先行 / 市場に前例なし | false |
| F6 | この機能に関する過去の経緯は？ | 過去の経緯 | 検討して見送り / 過去に実装済み / 初めての検討 / 外部要因で変更 | false |

- F4: knowledge/{product}/ 配下の既存featureフォルダ名から動的に選択肢を生成する

## Phase C: OST（Opportunity Solution Tree）構造化

Phase B ヒアリング完了後、収集した情報を Teresa Torres の Opportunity Solution Tree で構造化する。
これは追加ヒアリングではなく、**収集済み情報の整理ステップ**である。

### OST構造化プロセス

1. **Desired Outcome（期待成果）の特定**
   - Phase A の NSM / KPI 回答から導出
   - 例: 「提案品質の向上によるマッチング率改善」

2. **Opportunities（機会）の整理**
   - Phase B の F3（課題カテゴリ）と既存 known_issues から抽出
   - 各機会に対して「ユーザーがこれを解決したい理由」を1文で記述

3. **Solutions（ソリューション候補）の列挙**
   - Phase B の回答 + Knowledge の improvement_hypotheses から抽出
   - 各ソリューションを対応する機会に紐付ける

4. **Experiments（検証方法）の提案**
   - 各ソリューションの不確実性が高いものに対して検証方法を提案
   - prd-writer の前提条件マッピング（assumptions-mapping）への入力となる

### OST出力フォーマット

ドメインサマリーに `ost_structure` フィールドとして以下を追加:

```yaml
ost_structure:
  desired_outcome: "{期待成果}"
  opportunities:
    - opportunity: "{機会1}"
      why: "{理由}"
      solutions:
        - "{ソリューション1}"
        - "{ソリューション2}"
    - opportunity: "{機会2}"
      why: "{理由}"
      solutions:
        - "{ソリューション3}"
  high_uncertainty_items:
    - solution: "{ソリューション名}"
      suggested_experiment: "{検証方法}"
```

## ヒアリング進行ルール

1. **スキップ判定:** 各質問の前にKnowledgeの既存データをチェック。データがあれば確認パターンに変換
2. **深掘り判定:** 「Other」選択時や回答が曖昧な場合は1回だけフォローアップ質問を追加
3. **完了判定:** Phase A + Phase B の必須項目がすべて埋まったら完了。Phase C のOST構造化まで実施して終了
4. **中断対応:** ユーザーが「わからない」「後で」と回答した場合は [未回答] としてマークし、PRD生成時に [要確認] として反映
5. **エビデンス収集:** 課題・ペインに対して根拠（データ、URL、画像、ユーザーの声）を確認する。根拠がない場合は [仮説段階] としてマークする

## 出力フォーマット

ヒアリング完了後、以下の構造でサマリーを生成:

```yaml
domain_summary:
  product: "{プロダクト名}"
  feature: "{機能名}"
  feature_category: "{カテゴリ}"
  key_findings:
    - "{発見1}"
    - "{発見2}"
  ost_structure:
    desired_outcome: "{期待成果}"
    opportunities:
      - opportunity: "{機会}"
        why: "{理由}"
        solutions:
          - "{ソリューション}"
    high_uncertainty_items:
      - solution: "{ソリューション名}"
        suggested_experiment: "{検証方法}"
  knowledge_updates:
    - file: "{更新ファイルパス}"
      changes: "{変更内容}"
  evidence_items:
    - issue: "{課題}"
      evidence_type: "data|image|voice|unconfirmed"
      content: "{具体的な根拠内容/URL/ファイルパス}"
  unanswered:
    - "{未回答の質問}"
  context_for_next_step: "{次のステップに引き継ぐコンテキスト}"
```
