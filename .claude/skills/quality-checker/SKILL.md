# PRD品質チェッカー スキル

生成されたPRDの品質を検証するチェックリスト。
全項目 PASS で品質合格とする。

## チェックリスト

### 構造チェック（Structure）

- [ ] **S1:** 全14セクション＋参考リンク集に内容が記入されているか
- [ ] **S2:** 各セクションが prd-template/SKILL.md のフォーマットに準拠しているか
- [ ] **S3:** [要確認] タグが付いた項目が明示されているか

### 仮説チェック（Hypothesis）

- [ ] **H1:** 仮説が Who / Pain / Gain の構造になっているか
- [ ] **H2:** 各ステークホルダー（クライアント・ランサー・運営）の仮説が含まれているか
- [ ] **H3:** 仮説が検証可能な形で記述されているか（測定可能な指標があるか）
- [ ] **H4:** ペイン・課題に根拠（evidence）が記載されているか、または [仮説段階] タグが付いているか
- [ ] **H5:** リサーチ補足根拠がある場合、`[リサーチ補足]` タグ付きでユーザー提供根拠と区別されているか。また出典（source）が明記されているか

### ステークホルダーチェック（Stakeholder）

- [ ] **ST1:** 3者（クライアント・ランサー・運営）の視点が含まれているか
- [ ] **ST2:** ターゲットユーザーと課題が各ステークホルダーで記述されているか
- [ ] **ST3:** スコープ（作るもの）がステークホルダー別に整理されているか

### ユーザーストーリーチェック（User Story）

- [ ] **US1:** Why / What / できないこと を網羅しているか
- [ ] **US2:** INVEST基準を満たしているか（少なくとも主要ストーリー）
- [ ] **US3:** 受け入れ基準（Acceptance Criteria）が定義されているか

### 要件チェック（Requirements）

- [ ] **R1:** 要件に [must] / [want] の優先度が付いているか
- [ ] **R2:** 各要件が具体的で実装可能なレベルか
- [ ] **R3:** 「作らないもの」が明示されているか

### 指標チェック（Metrics）

- [ ] **M1:** KPI に具体的な数値目標があるか
- [ ] **M2:** NSM に具体的な数値目標があるか
- [ ] **M3:** 「価値を産んだ状態」が KPI/NSM と整合しているか

### リスクチェック（Risk）

- [ ] **RK1:** リスクに対して軽減策が1:1で定義されているか
- [ ] **RK2:** リスクが Tigers / Paper Tigers / Elephants に分類されているか（推奨）
- [ ] **RK3:** 影響度と発生確率が評価されているか

### 撤退・計画チェック（Exit & Plan）

- [ ] **E1:** 撤退基準が定量的な条件で記述されているか
- [ ] **E2:** リリース計画が MVP → Phase の段階的構成か
- [ ] **E3:** 各フェーズに成功指標が定義されているか

### スコープチェック（Scope）

- [ ] **SC1:** 「作るもの」と「作らないもの」が明確か
- [ ] **SC2:** Phase1 のスコープが適切なサイズか（大きすぎ/小さすぎないか）
- [ ] **SC3:** 「作らないもの」に理由が記載されているか

### 前提条件チェック（Assumptions）

- [ ] **AM1:** 前提条件マッピングが作成されており、3件以上の前提条件が抽出されているか
- [ ] **AM2:** 各前提条件に Impact × Confidence の評価が付いているか
- [ ] **AM3:** 🔴最優先検証の前提条件に、具体的な検証アクションと推奨期限が記載されているか

## 品質判定

### 合格基準
- **全項目 PASS:** 品質合格 → PRD出力
- **FAIL あり（自動修正可能）:** 自動修正を実施してから出力
- **FAIL あり（自動修正不可）:** [要確認] タグを付けてユーザーに提示

### 自動修正が可能なケース
- [must] / [want] の付け忘れ → コンテキストから推定して付与
- 仮説のWho/Pain/Gain構造の不備 → 既存テキストから再構造化
- 軽減策の欠落 → 一般的な軽減策を提案（[要確認]付き）
- エビデンスタグの欠落 → [仮説段階] タグを自動付与
- リサーチ補足根拠に `[リサーチ補足]` タグがない → タグを自動付与

### 自動修正が不可能なケース
- KPI/NSMの数値目標が不明 → ユーザーに確認が必要
- ステークホルダーの視点が完全に欠落 → ヒアリング情報が不足
- 撤退基準の数値 → ビジネス判断が必要

## Pre-Mortem リスク分類ガイド（pm-skills pre-mortem より）

PRD品質チェックのリスクセクション（RK1〜RK3）では、以下のPre-Mortem手法を活用:

### 思考プロセス
1. プロダクトが14日後にリリースされると想像する
2. 次に、それが失敗したと想像する — ユーザーが採用せず、収益目標を達成せず、評判が傷ついた
3. 「何がうまくいかなかったか？」「何を見落としたか？」「何に自信過剰だったか？」と問う

### 3分類の詳細

**Tigers（本物のリスク）:**
- 証拠、過去の経験、明確なロジックに基づくリスク
- 夜も眠れなくなるようなもの
- アクションが必要
- 緊急度分類:
  - **Launch-Blocking:** リリース前に解決必須
  - **Fast-Follow:** リリース後30日以内に解決
  - **Track:** リリース後モニタリングし、問題化したら対処

**Paper Tigers（見かけ倒しのリスク）:**
- 表面上は妥当な懸念だが、実際には起こりにくいか影響が小さい
- 大きなリソース投資に値しない
- ステークホルダーの認識合わせのために文書化

**Elephants（誰も話さないリスク）:**
- 誰も検証していない暗黙の懸念や前提
- 本物かもしれないが、確信が持てない
- リリース前に調査が必要

### Launch-Blocking Tiger のアクションプラン
各 Launch-Blocking Tiger について:
- リスクの明確な記述
- 具体的な軽減アクション
- 最適なオーナー（機能/担当者）
- 判断/完了期限
