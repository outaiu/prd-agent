# ユーザーストーリー スキル

高品質なユーザーストーリーを作成するためのフレームワーク。
3C's と INVEST 基準に基づく。

## 3C's フレームワーク

### 1. Card（カード）
ストーリーの簡潔な記述。以下の形式:

```
As a [ユーザータイプ],
I want [達成したいこと],
So that [理由・価値]
```

日本語PRDでの対応形式:

**Why（動機）:**
```
〇〇は、〇〇の時に、〇〇をしたい。なぜならば〇〇だからだ。
```

**What（実現すること）:**
```
〇〇が、〇〇で、〇〇出来ること。
```

### 2. Conversation（対話）
- ストーリーの詳細を明確にするための対話内容
- 前提条件、ビジネスルール、制約
- 質問事項と回答の記録
- エッジケースの特定

### 3. Confirmation（確認）
- 受け入れ基準（Acceptance Criteria）
- Given-When-Then 形式:

```
Given: [前提条件]
When: [アクション]
Then: [期待結果]
```

## INVEST 基準

各ユーザーストーリーが以下を満たすことを確認:

| 基準 | 説明 | チェック方法 |
|------|------|-------------|
| **I**ndependent | 他のストーリーから独立して実装可能 | 依存関係がないか確認 |
| **N**egotiable | 詳細は交渉可能（実装方法を規定しない） | How を含まず What に焦点 |
| **V**aluable | ユーザーまたはビジネスに価値を提供 | So that が明確か |
| **E**stimable | 工数の見積もりが可能 | スコープが明確か |
| **S**mall | 1スプリント以内で完了可能なサイズ | 分割が必要でないか |
| **T**estable | テスト可能な受け入れ基準がある | Given-When-Then が書けるか |

## ストーリー作成プロセス

### Step 1: ペルソナ特定
- Knowledge のステークホルダー・ペルソナ情報を参照
- PRD対象機能に関連するペルソナを選定

### Step 2: ストーリーマッピング
ユーザーの行動フローに沿ってストーリーを配置:

```
ユーザーアクティビティ（大きな目標）
├── ステップ1（サブゴール）
│   ├── ストーリー1.1
│   └── ストーリー1.2
├── ステップ2
│   ├── ストーリー2.1
│   └── ストーリー2.2
└── ステップ3
    └── ストーリー3.1
```

### Step 3: 優先度付け
- [must] ストーリー: MVP/Phase1 に必須
- [want] ストーリー: あると望ましい
- MoSCoW法（Must/Should/Could/Won't）も活用可能

### Step 4: 受け入れ基準作成
各ストーリーに Given-When-Then を付与

### Step 5: 「できないこと」の定義
- 意図的にサポートしないユースケース
- 技術的制約による制限
- 将来フェーズに先送りする機能

## PRDユーザーストーリーセクションとの対応

PRDの「9. ユーザーストーリー」セクションには以下を記述:

### Why（動機）
- 3C's の Card 部分
- ペルソナ × 機能的/感情的/社会的 Job の組み合わせ

### What（実現すること）
- ストーリーマッピングの各ストーリー
- [must] / [want] の優先度付き

### できないこと（制約条件）
- Step 5 で定義した除外事項
- 理由を明記

## ストーリー分割のパターン

大きすぎるストーリーを分割する際のパターン:

1. **ワークフローステップ別:** 各ステップを独立したストーリーに
2. **ビジネスルール別:** 各ルールを独立したストーリーに
3. **データバリエーション別:** 入力パターンごとに分割
4. **操作別:** CRUD（作成/読取/更新/削除）に分割
5. **ペルソナ別:** 各ペルソナのストーリーを分離
6. **ハッピーパス/エッジケース別:** 正常系と異常系を分離

## ストーリーテンプレート例（pm-skills user-stories より）

### 記述形式

**Title:** [機能名]

**Description:** As a [ユーザーロール], I want to [アクション], so that [ベネフィット].

**Design:** [デザインファイルへのリンク]

**Acceptance Criteria:**
1. [明確でテスト可能な基準]
2. [観察可能な振る舞い]
3. [システムが正しくバリデーション]
4. [エッジケースのハンドリング]
5. [パフォーマンスまたはアクセシビリティの考慮]
6. [インテグレーションポイント]

### 具体例

**Title:** 最近閲覧したセクション

**Description:** オンラインショッパーとして、商品ページに「最近閲覧した」セクションを表示して、検討した商品に簡単に戻りたい。

**Acceptance Criteria:**
1. 過去に1つ以上の商品を閲覧したユーザーに「最近閲覧した」セクションが商品ページ下部に表示される
2. セッションで最初の商品ページを訪問するユーザーには表示されない
3. 現在表示中の商品自体はリストから除外される
4. 商品カードに画像、タイトル、価格が表示される
5. 各商品カードに閲覧時刻が表示される（例:「5分前に閲覧」）
6. 商品カードをクリックすると対応する商品ページに遷移する

## 成果物チェックリスト
- [ ] 機能の完全なユーザーストーリーセット
- [ ] 各ストーリーにタイトル、説明、デザインリンク、4〜6個の受け入れ基準
- [ ] ストーリーは独立しており、どの順序でも開発可能
- [ ] ストーリーは1スプリントサイクルに収まるサイズ
- [ ] 関連デザインドキュメントへの参照
